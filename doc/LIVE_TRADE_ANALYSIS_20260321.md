## 2. 根因（代码级）

### 2.1 `MarketContext.signal_confidence` 与 `"Up"` 字符串

- `Direction` 枚举值为 `"Up"` / `"Down"`。
- `market_context.py` 曾使用 `signal_dir == "UP"`，导致 **`"Up"` 被误判为「做空方向」**，下单日志里的「可信=0.32」与心跳「可信=0.67」**同时出现**。
- **策略实际传入**的是 `mc_conf_up`（来自正确调用 `"UP"`），因此 **策略未用错分**；但 **Telegram/日志展示严重误导**，且任何未来若用 `signal.direction.value` 参与过滤都会出错。

**修复**：`signal_up = (signal_dir.upper() == "UP")`。

### 2.2 `bot.py` 中「逆势 conf 抬升到 ≥0.55」

- 原意：回测中「赔率与 30m 趋势反向」时胜率不低，故把 `mc_conf_up`/`mc_conf_dn` **抬到至少 0.55** 再交给策略。
- **副作用**：分2 **Down** 在 **30m 上涨** 环境下，原始可信约 **0.28**，抬升后变成 **≥0.55**，**绕过** `_eval_odds_only` 里 **分2 Down 的 conf≥0.35** 与后续 EV 门槛，导致 **#326、#327** 类逆势空单仍成交。

**修复**：取消抬升，策略只使用 **原始** `res_up["score"]` / `res_dn["score"]`。

### 2.3 分2 Down：强赔率例外

取消抬升后，**仅**用 `conf≥0.35` 会 **误杀** 早期成功案例（如 **02:32** Down **0.865**、raw conf **0.28**）。

**修复**：分2 Down 采用分层：

- 默认 **`signal_confidence ≥ 0.35`**；
- 若 **`odds_px ≥ 0.85` 且 gap 与 Down 同向**，允许 **`≥ 0.28`**（与高赔率实盘赢家一致）。

这样 **0.79 / 0.82** 且 conf **0.28** 的逆势单会被拦截，**0.865+** 同向仍可通过。

### 2.4 路径2 剩余时间过短

- **#325**：`剩余31s` 仍触发路径2，随后数秒内赔率剧烈摆动。
- **修复**：路径2 增加 **`PATH2_MIN_SECS_REMAINING = 40`**（剩余不足 40s 不再下路径2）。
