# 第二阶段：历史数据辅助模块

> **版本 1.0** — 基于 Phase 1 实盘数据采集经验总结（2026-03-17）

---

## 一、背景与结论概述

### 1.1 问题来源

在 Phase 1 数据采集过程中，观察到末期套利信号存在以下误判场景：

| 窗口时间 | 问题 | 结果 |
|---------|------|------|
| 16:20 窗口 | BTC 强势上涨，但市场说 DOWN=0.65，跟赔率做 DOWN | 赔率可能错误，BTC 最终接近 UP |
| 16:46 窗口 | PtB 延迟87s+赔率静止，系统误报多次强套利 | 信号不可靠，已修复 |
| 01:40 窗口 | 市场静止偏置 Down=0.54，软冲突阻止了 UP 信号 | 最终 **UP 胜出**，信号被错误阻止 |

**共同特征**：如果在信号判断时能了解"过去 30–90 分钟 BTC 的趋势背景"，这些误判中的大部分可以被正确处理。

### 1.2 核心结论

> **历史数据无法可靠预测 5 分钟涨跌方向，但可以显著提升现有末期套利信号的质量。**

| 用途 | 可行性 | 预期收益 |
|------|-------|---------|
| 单独预测5分钟涨跌方向 | ❌ 胜率仅52–54%，边际极薄 | EV ≈ +0.03/美元 |
| 过滤/增强末期套利信号 | ✅ 减少误判，提高信号可靠性 | 降低假正例率 ~30% |
| 预测 gap 机会频率 | ✅ 波动率聚集效应可用 | 优化入场时机 |
| 评估信号趋势一致性 | ✅ 顺势信号比逆势信号更可靠 | 提高有效信号比例 |

## 二、为什么不能单独用历史数据预测方向

### 2.1 市场已将公开历史信息定价

```
若 BTC 过去60分钟持续上涨 +0.8%，所有市场参与者都能看到。
专业套利者会将 Up 赔率推高到 0.55–0.60。

此时即使你的模型说"趋势延续概率 53%"：
EV = 0.53 × (1-0.57) - 0.47 × 0.57 = 0.228 - 0.268 = -0.040  ← 亏损
```

**历史趋势是公开信息，Polymarket 赔率会实时反映它，套利空间几乎为零。**

### 2.2 数学极限对比

| 方法 | 理论胜率 | 在赔率0.51时的 EV | 备注 |
|------|---------|-----------------|------|
| 随机猜测 | 50% | -0.010 | 基准线 |
| 30-90分钟趋势分析 | 52–54% | +0.010–+0.030 | 边际极薄 |
| **末期套利（当前策略）** | **89–97%** | **+0.387–+0.463** | **高15倍以上** |

### 2.3 BTC 5分钟价格的随机游走特性

来自 `backtest.py` 的实证结论（7天 × 2015个窗口）：

- 动量策略（过去60秒方向）：胜率 **46.1%**（低于随机！）
- 均值回归策略：胜率 **50.5%**（无显著优势）
- 结论：**BTC 5分钟价格变动接近随机游走**，任何基于历史价格走势的方向预测效果极差。

## 三、历史数据的三个有效用途

### 用途A：趋势一致性过滤器（信号质量评分）

**原理**：末期套利信号如果与大趋势方向一致，可靠性更高；如果逆势，误判概率更高。

```python
# 示例逻辑（伪代码）
trend_90m = calc_trend(btc_candles_90m)   # +1 上涨趋势，-1 下跌趋势，0 震荡

signal_direction = "UP"  # 当前末期套利信号

if signal_direction == "UP" and trend_90m > 0:
    confidence = "HIGH"    # 顺势信号，可靠
elif signal_direction == "UP" and trend_90m < 0:
    confidence = "LOW"     # 逆势信号，需要更大 gap 阈值才入场
```

**对应的实盘案例**：
- 16:20 窗口：BTC 强势上涨（趋势 UP），市场却给出 DOWN=0.65 信号 → 逆势赔率，应降低可信度
- 01:40 窗口：BTC 震荡（趋势中性），市场给出 Down=0.54 → 仅为轻微静止偏置，不应阻止 UP 信号

### 用途B：波动率过滤器（预测机会频率和质量）

**原理**：波动率聚集效应（ARCH 效应）——高波动率时段后，gap 更大，套利机会更多且更可靠。

```python
# 过去30分钟1分钟K线的真实波动率（ATR）
volatility_30m = calc_atr(btc_1m_candles[-30:])

if volatility_30m > threshold_high:
    # 高波动时段：gap 更大，信号更强，可使用正常仓位
    position_multiplier = 1.0
elif volatility_30m < threshold_low:
    # 低波动时段：gap 小，信号弱，减小仓位或提高阈值
    position_multiplier = 0.5
    min_gap_threshold = 0.15  # 提高门槛
```

**关键指标定义**：

| 指标 | 计算方式 | 用途 |
|------|---------|------|
| ATR-30m | 过去30分钟每分钟 high-low 均值 | 当前波动率水平 |
| Vol-ratio | ATR-30m / ATR-90m | 波动率是否在扩大 |
| Gap-prob | 历史同等波动率下触发0.10% gap的概率 | 预测本窗口是否会有信号 |

### 用途C：动态 PtB 偏差估算

**原理**：CC 估算的 PtB 与真实 Chainlink PtB 的偏差，在强趋势中更大（Chainlink 滞后于快速行情）。

```python
# 过去5分钟BTC涨幅越大，Chainlink PtB可能比CC估算高/低越多
recent_5m_change = (btc_price_now - btc_price_5m_ago) / btc_price_5m_ago * 100

# 强上涨时，Chainlink PtB可能高于CC估算（实际gap比估算小）
# 强下跌时，Chainlink PtB可能低于CC估算（实际gap比估算小）
ptb_bias_correction = recent_5m_change * 0.3  # 经验系数，需实盘校准

adjusted_gap = estimated_gap - ptb_bias_correction
```

**数据支撑**：Phase 1 采集显示 CC 与真实 PtB 偏差范围：
- 平静市场：$20–$50（0.03–0.07%）
- 快速行情：$100–$150（0.13–0.20%）

## 四、实施方案

### 4.1 数据来源

```python
# 已有 Binance 1分钟K线数据（实时）
# 新增：从 Binance 获取过去 90 分钟的历史 K 线
BINANCE_KLINES_URL = (
    "https://api.binance.com/api/v3/klines"
    "?symbol=BTCUSDT&interval=1m&limit=90"
)
```

API 返回字段：`[open_time, open, high, low, close, volume, ...]`

免费接口，无需 API Key，与现有 Binance 价格接口同源。

### 4.2 计算模块设计

```
scripts/
  collect_data.py        ← 现有：实时采集（每5秒一次）
  analyze_collected.py   ← 现有：离线分析
  
src/
  market_context.py      ← 新增：历史数据辅助模块
```

**`src/market_context.py` 核心函数**：

```python
class MarketContext:
    """
    每个5分钟窗口开盘时更新一次，提供历史背景信息。
    """
    def fetch_klines(self, n_minutes: int = 90) -> list[dict]:
        """从 Binance 获取最近 n 分钟的1分钟K线"""
        ...

    def calc_trend(self, window_minutes: int = 30) -> float:
        """
        计算趋势强度：+1.0（强上涨）到 -1.0（强下跌）
        基于：线性回归斜率 + 收盘价在均线上下的比例
        """
        ...

    def calc_volatility(self, window_minutes: int = 30) -> float:
        """
        计算近期波动率（ATR），单位：%
        """
        ...

    def signal_confidence(
        self,
        signal_dir: str,    # "UP" or "DOWN"
        gap_pct: float,     # 当前 gap
        minute: int,        # 窗口内分钟数
    ) -> dict:
        """
        综合评分，返回：
        {
          "score": 0.0–1.0,       # 信号可信度
          "trend_aligned": bool,   # 是否顺势
          "volatility_level": str, # "high"/"medium"/"low"
          "recommended_threshold": float,  # 建议的 gap 门槛
          "note": str              # 说明文字
        }
        """
        ...
```

### 4.3 与现有策略的集成方式

**不改变核心信号逻辑**，只在信号输出时叠加一层评分：

```
当前流程：
  末期套利信号 → 🟢 强套利! 方向=DOWN EV=+0.543

集成后流程：
  末期套利信号 + 历史背景评分 → 🟢 强套利! 方向=DOWN EV=+0.543 [趋势顺势↓ 波动率高 可信度:0.87]
                                → ⚠️ 注意逆势! 方向=DOWN EV=+0.543 [趋势上涨↑ 建议gap门槛提高到0.15%]
```

### 4.4 分阶段实施计划

**阶段一（数据积累期，当前 Phase 1）**：
- 在 `observations.db` 中同步记录每次信号时的 90 分钟历史背景数据
- 不改变任何信号逻辑，只是记录额外字段
- 目标：积累 500+ 个有信号的窗口样本

**阶段二（回归分析期）**：
- 分析已记录的数据：趋势背景 vs 实际结果的相关性
- 确定趋势过滤的最优阈值（避免过拟合）
- 估算 CC PtB 偏差的修正系数

**阶段三（集成优化期）**：
- 将 `MarketContext` 模块集成到 `collect_data.py` 和 `src/bot.py`
- A/B 测试：有历史背景过滤 vs 无历史背景过滤的信号质量差异
- 根据实盘结果调整权重

## 五、预期效果与风险

### 5.1 预期改善

| 指标 | 当前（无历史背景） | 集成后（有背景过滤） | 改善幅度 |
|------|-----------------|-------------------|---------|
| 假正例率（信号错误） | 估算 ~25% | 目标 ~15% | 减少 40% |
| 顺势信号比例 | 未统计 | 目标 > 70% | — |
| 逆势信号误判率 | 高 | 降低 | 定量待实测 |

### 5.2 主要风险

1. **过拟合风险**：历史数据中找到的规律可能在未来失效
   - 缓解：保持简单线性指标（ATR、斜率），避免复杂模型

2. **增加延迟**：每窗口多一次 API 调用（Binance K 线）
   - 缓解：在窗口开盘时预取，缓存5分钟，不影响信号实时性

3. **过度过滤**：趋势过滤器可能把真正有效的逆势信号也过滤掉
   - 缓解：逆势信号不是"阻止"而是"提高门槛"（gap 从 0.10% 提高到 0.15%）

4. **参数敏感性**：90分钟窗口、趋势阈值等参数需要实盘数据来校准
   - 缓解：阶段一只记录数据，阶段二才确定参数

### 5.3 不应做的事

- ❌ 不用历史数据**替代**末期套利作为主要入场逻辑
- ❌ 不用机器学习模型来预测方向（样本量不足、过拟合风险高）
- ❌ 不根据趋势单独做方向性下注（EV 太低，已验证无效）
